# Secure Inventory Tracer Bullet Spec

## Purpose

Stuff Stash needs its first vertical slice to prove security boundaries before product behavior grows.

This tracer bullet must show that an HTTP request can move through authentication, authorization, application services, domain rules, observability, and OpenAPI-backed responses.

## Scope

This spec covers the first implemented secure API slice:

- Local development authentication.
- Authorization boundaries.
- Tenant creation.
- Inventory creation.
- Listing authorized inventories.
- Current principal lookup.
- Huma REST/OpenAPI wiring.

This spec does not implement Google OIDC, assets, locations, or generated client SDKs.

The next auth/authz tracer bullet extends this slice by adding production-shaped OIDC authentication, SpiceDB authorization, and GORM/Postgres persistence adapters behind the same ports.

## Decisions

- The first authentication implementation is a local development adapter behind the authentication port.
- Local development authentication must use deterministic bearer tokens for tests and local runs.
- The first authorization implementation is an in-memory relationship adapter behind the authorization port.
- A SpiceDB schema file must exist before the production SpiceDB adapter is implemented.
- Huma must be used for the first REST endpoints and generated OpenAPI.
- Persistence starts in-memory for fast tests and local runs.
- Compose must use the GORM/Postgres repository adapter.
- The repository contract must be verified with a real GORM-backed fake.

## Local Development Authentication

- Local dev requests must use `Authorization: Bearer dev:<principal-id>`.
- The principal ID must be validated and treated as an authenticated user ID.
- Missing, malformed, or unsupported tokens must be rejected.
- The local dev adapter must not be treated as a production OIDC adapter.
- The authentication port must be shaped so Google OIDC can be added later without changing application services.

## Authorization

- Authorization must be relationship-based.
- Tenant creators become tenant owners.
- Inventory creators become inventory owners.
- Tenant owners may create inventories inside their tenant.
- Users may list only inventories they can view.
- Inventory owners may view their inventory.
- Authorization checks must happen inside application services, not only in HTTP handlers.
- Authorization failures must use safe error responses.

## Endpoints

All endpoints except `GET /healthz` require authentication.

The first endpoints are:

- `GET /me`
- `POST /tenants`
- `POST /tenants/{tenantId}/inventories`
- `GET /tenants/{tenantId}/inventories`

Responses must follow the project API envelope rules.

## Security Tests

Every protected endpoint must have adversarial tests for:

- Missing authorization header.
- Malformed authorization header.
- Unsupported token scheme.
- Cross-tenant access.
- Valid authorized access.
- Valid authenticated but unauthorized access where applicable.
- Tenant owners must be able to list all inventories in their tenant.
- Inventory owners who are not tenant owners may list the tenant, but must only see inventories they can view.
- Authenticated users without any relationship to a tenant or inventory must not list that tenant's inventories.
- Authenticated users without tenant create-inventory permission must not create inventories in that tenant.
- Hidden or unauthorized resources must use safe error responses that do not expose internal details.

## Observability

- Authentication failures must be observable without logging credentials.
- Authorization failures must be observable without leaking hidden resources.
- Tenant and inventory creation must be observable.

## OpenAPI

- OpenAPI must be generated by Huma from registered operations.
- OpenAPI must include the first secure endpoints.
- OpenAPI must expose bearer authentication for protected endpoints.

## Out Of Scope

- Production Google OIDC.
- Production SpiceDB client adapter.
- PostgreSQL persistence.
- Database migrations.
- Generated TypeScript SDKs.
